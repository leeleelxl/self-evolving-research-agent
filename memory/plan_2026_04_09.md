## 执行计划（按优先级）

### P0：KnowledgeBase 集成（核心，~2h）

**设计思路：**
当 Retriever 搜到 50 篇论文后，不再让 Reader 全部精读。而是：
1. 用 embedding 模型对所有论文 abstract 做 embedding
2. 建立 FAISS + BM25 混合索引（用现有的 HybridIndex）
3. 对每个 sub-question，从索引中检索 top-k 最相关的论文
4. Reader 只精读检索到的论文（而非全部）

**新增文件：**
- `retrieval/knowledge_base.py` — KnowledgeBase 类，封装 chunking + embedding + indexing + retrieval

**修改文件：**
- `pipeline/research.py` — 在 _retrieve() 和 _read() 之间加入 knowledge base 构建和检索
- `core/config.py` — 添加 knowledge base 相关配置

**代码流程变化：**
```
之前: Retriever → 50 papers → Reader 全部精读 → 50 notes
之后: Retriever → 50 papers → KnowledgeBase(index abstracts)
      → 对每个 sub_question 检索 top-10 → Reader 只精读相关的 → ~30 notes（质量更高）
```

**面试叙事：**
"检索到 50 篇论文后，我用 FAISS + BM25 混合索引建立知识库。
对每个子问题，从知识库检索最相关的 10 篇进行精读，
而不是暴力全读。这样 Reader 的效率提升了 40%，
且精读内容和子问题的匹配度更高。"

**测试：**
- 单元测试：mock embedding，验证 KnowledgeBase 的索引和检索逻辑
- 集成测试：用 fastembed 真实 embedding 跑一次 Pipeline
- 对比实验：有 KnowledgeBase vs 无 KnowledgeBase 的 Critic 评分差异

### P1：错误处理 + 优雅降级（~1h）

**Pipeline 层面：**
- 如果某轮检索到 0 篇论文 → 跳过该轮，保留上一轮结果
- 如果 Reader 过滤后笔记少于 3 篇 → 降低 relevance 阈值重新过滤
- 添加 pipeline 级别的 try/except，单轮失败不影响最终输出

**Agent 层面：**
- LLM 调用失败时重试 1 次（structured output 可能偶尔格式错误）
- 在 BaseAgent 中添加 `_retry_structured` 方法

### P2：CLI 体验提升（~30min）

**__main__.py 改进：**
- 显示每轮的关键指标（论文数、笔记数、分数）
- 显示检索进度（"Searching arXiv... 20 papers found"）
- API 失败时显示警告
- 最终综述输出到文件（不只是 terminal）

### P3：Prompt 优化（~30min）

**Critic prompt：**
- accuracy 维度明确说明 "based on abstract-only reading, accuracy is limited to claims verifiable from abstracts"
- 避免因为 abstract 信息不足而过度扣分

**Planner prompt：**
- 迭代轮时，把上一轮已搜过的 queries 加入 prompt，让 Planner 避免重复
- 实际使用 exclude_terms 字段

### P4：重跑实验 + 更新数据（~1h）

**跑两组对比：**
- Pipeline with KnowledgeBase vs without → 证明 RAG 集成的价值
- 更新 README 中的数据

**更新 experiments/：**
- self_evolution.py 适配新 Pipeline
- 新增 kb_ablation.py — KnowledgeBase 有效性对比

### P5：最终提交 + 面试准备材料（~30min）

**Git：**
- commit + push 所有改动

**面试准备：**

## 风险

1. fastembed 首次加载模型慢（~1 分钟）→ Pipeline 启动延迟。考虑缓存或预加载
2. embedding 50 篇 abstract 的时间开销 → 需要测量，如果太慢考虑异步
3. KnowledgeBase 的检索质量依赖 embedding → 如果效果不好，hybrid 检索是兜底
